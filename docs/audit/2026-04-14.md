# Security Audit — 2026-04-14

> **Scope**: bote MCP service (Cyrius port) — all six transports
> (stdio / HTTP / Unix / bridge / streamable HTTP / WebSocket), the
> auth + content + host modules, codec + jsonx + session, and the
> libro_tools surface. **Trigger**: 2.0 prep — explicit "0-day and
> CVE external research and repair" request.
>
> **Method**: external CVE-class survey driven by a security-focused
> sub-agent + manual source review at every flagged site. No outbound
> network traffic; classes drawn from the auditor's existing CVE
> knowledge (HTTP smuggling, header injection, timing oracles, JSON
> DoS, request-size limits, session-ID predictability, SSRF bypass
> vectors, error-message disclosure, memory safety on bump-allocator).
>
> **Result**: 14 findings (3 critical, 5 high, 5 medium, 1
> informational). 9 fixed in 1.9.4 + 1.9.5; 5 deferred to 2.0 / cyrius
> upstream.

## Findings

### Critical (3)

| ID | Finding | File:Line | Status |
|---|---|---|---|
| **C1** | **SSRF bypass — integer-form IPv4.** `http://2130706433/` (the integer encoding of 127.0.0.1) parses as a 0-dot "hostname"; `_ssrf_classify_ipv4` returned `SSRF_PARSE`, falling through to `_ssrf_classify_hostname` which only blocked literal `localhost` / `metadata`. Result: glibc / DNS resolver treated `2130706433` as integer IPv4 → loopback fetch reached. | `src/host.cyr:159–204` | ✅ Fixed in **1.9.5** — hostname classifier now rejects all-`[0-9.]` hosts as `SSRF_PARSE`. |
| **C2** | **SSRF bypass — octal IPv4.** `_ssrf_parse_octet` accepted leading-zero multi-digit forms: `0177` parsed as decimal 177 (apparently public) while glibc `inet_aton` interprets it as octal 127 → loopback. Hex (`0x7f`) was already rejected because `x` isn't `[0-9]`. | `src/host.cyr:136–153` | ✅ Fixed in **1.9.5** — octet parser now rejects digits>1 with leading zero (`00`, `01`, `0177`, `010`). |
| **C3** | **SSRF bypass — IPv4-mapped IPv6.** `http://[::ffff:127.0.0.1]/` and `http://[::ffff:7f00:1]/` were classified by the IPv6 prefix-blocklist, which only matched `::1`, `::`, `fe80:`, `fc/fd`, `ff` exact. The `::ffff:` IPv4-mapped prefix passed as `SSRF_OK`. Connect would land on loopback. **Plus**: `_ssrf_classify_ipv4` didn't verify the parser consumed all bytes between dots — `64:ff9b::1.2.3.4` parsed as `64.2.3.4` (public) and never reached the IPv6 path. | `src/host.cyr:224–258`, `src/host.cyr:181–184` | ✅ Fixed in **1.9.5** — IPv6 classifier now blocks `::ffff:`, `::*.*.*.*` (v4-compatible), `64:ff9b:` (NAT64 well-known) outright. IPv4 classifier now requires each octet parse to consume exactly the dot-to-dot range. |

### High (5)

| ID | Finding | File:Line | Status |
|---|---|---|---|
| **H1** | **HTTP request smuggling — Transfer-Encoding never parsed.** `http_recv_request` (in stdlib `lib/http_server.cyr`) reads only `Content-Length`. A front-proxy that honors TE and forwards to bote (which only honors CL) is the canonical CL.TE smuggling setup (CVE-2019-18276 family). Connection-close limits weaponization today, but any keep-alive future would open it. | `lib/http_server.cyr:389–418`, `src/transport_http.cyr`, `src/transport_streamable.cyr`, `src/bridge.cyr`, `src/transport_ws.cyr` | ✅ Fixed in **1.9.4** — every HTTP-family transport now rejects any inbound `Transfer-Encoding` with 400 (3-line inline check, no shared helper because of cyrius identifier-table headroom). |
| **H2** | **Timing-attack on bearer-token comparison.** `auth_validator_allowlist` used `streq` which short-circuits on first byte mismatch. Network-observable per-byte timing lets an attacker guess a token byte-by-byte against a public bearer endpoint. | `src/auth.cyr:89–98` | ✅ Fixed in **1.9.4** — comparison is now constant-time: lengths checked first (token length still leaks; unavoidable for variable-length bearers), then bytes XOR-accumulated with no early exit. Outer vec walk also doesn't short-circuit on match (defeats the position-of-match oracle). 9 new test assertions. |
| **H3** | **JSON-RPC batch amplification DoS.** No cap on batch length. A 1 MiB body of `[{},{},...]` (~300k 3-byte elements) forces per-element response allocation; bump allocator never frees → OOM. | `src/codec.cyr:149–178` | ✅ Fixed in **1.9.4** — hard cap `n > 100` returns `-32600` "batch too large". 100 is generous (real clients batch single-digit calls). |
| **H4** | **No JSON-nesting depth cap.** `_jx_skip_struct` is iterative (no stack overflow), but with the 64 KB inbound buffer there's no defense if anyone raises that buffer; a 64 KB-deep `[[[[…]]]]` walks the structure linearly. CPU DoS. | `src/jsonx.cyr:39–72` | ✅ Fixed in **1.9.4** — `depth > 64` returns end-of-buffer. |
| **H5** | **Slowloris.** `http_server_run` serializes the accept loop per transport instance. A single connection that sends 1 byte and holds open blocks every other client until peer-disconnect or TCP keepalive. Trivial DoS against any HTTP-family transport. | `lib/http_server.cyr:450–460` | ⏳ Deferred to **2.0** — needs a `sock_set_recv_timeout` helper that doesn't exist in the cyrius `lib/net.cyr` stdlib yet. Workaround: deploy behind a reverse proxy (nginx / caddy) that handles slow connections. Tracked. |

### Medium (5)

| ID | Finding | File:Line | Status |
|---|---|---|---|
| **M1** | **Request size silently truncated.** `HSV_REQ_BUF_SIZE = 65536`. A 64 KB request that's all-headers (no body) causes `http_body_offset` to fail and the request is processed as malformed (400) instead of returning 413 Payload Too Large. Also: `_hsv_req_buf` is process-global → exactly one concurrent request ever (linked to H5). | `lib/http_server.cyr:426, 437, 453–454` | ⏳ Deferred — needs an explicit per-handler max-body cap with 413 response. Mostly a UX issue (current behaviour is "reject malformed", not "process attacker payload"). |
| **M2** | **Session-ID `/dev/urandom`-fallback was predictable.** If `/dev/urandom` open failed, sid was derived from `clock_now_ns()` ×2 with FNV prime (predictable to any timing observer). Also `syscall(SYS_READ, fd, raw, 16)` return ignored — a short read left bump-allocator garbage in the SID material. | `src/session.cyr:68–88` | ✅ Fixed in **1.9.4** — open-failure / short-read both write a fatal-error message and `SYS_EXIT(90)`. There is no acceptable substitute for a CSPRNG when minting session IDs. Read-loop guarantees 16 bytes. |
| **M3** | **Bridge CORS oracle.** `_bridge_cors_origin` returns `vec_get(allowed_origins, 0)` (first allowed origin) on a miss. Some CORS middlewares treat the echoed value as an oracle for "what origins does this server accept". | `src/bridge.cyr:158–186` | ⏳ Deferred to **2.0** — should omit `ACAO` entirely on miss or return the literal string `"null"`. Browser will block the request either way; informational leak. |
| **M4** | **WS handshake doesn't validate `Sec-WebSocket-Key` length.** RFC 6455 §4.1 requires base64-encoded 16 random bytes (24 chars). Stdlib `lib/ws_server.cyr` accepts any length and concatenates with magic + SHA-1 hashes. A 4 GiB key would `alloc(concat_len)` and OOM. | `lib/ws_server.cyr:191–230` | ⏳ Deferred to **2.0** — stdlib fix; bote can't patch it without forking. Track upstream. |
| **M5** | **Bridge skips protocol-version + session-ID checks.** Unlike `transport_http` and `transport_streamable`, the bridge `/` POST handler doesn't run `validate_protocol_version` or `_http_check_session`. By design (bridge is single-app-scoped), but a caller with a bearer token can submit any method including `initialize` without protocol negotiation. | `src/bridge.cyr:201–261` | ⏳ Deferred to **2.0** — add optional `bridge_config_with_require_protocol(c, 1)` and gate. |

## Trade-offs taken in 1.9.4

To fit the inlined security checks under cyrius 4.7.1's identifier-table
headroom, **`libro_tools` is no longer registered by default** — it became
opt-in via `BOTE_LIBRO=1` env var. Consumers who use the five
`libro_*` MCP tools just set `BOTE_LIBRO=1` (or include + register the
module manually). Will revert to default-on when cyrius 4.8.0 raises
the cap further; tracked.

## Test coverage added

- **9 auth assertions** covering constant-time compare: first-byte-differ vs last-byte-differ vs shorter vs longer (all rejected); position-independence across a 3-entry allowlist.
- **11 host assertions** covering SSRF bypasses: integer IPv4, octal IPv4, octal-looking IPv4, short-form IPv4 (`127.1`), `::ffff:127.0.0.1`, `::ffff:7f00:1`, `::127.0.0.1`, NAT64 `64:ff9b::1.2.3.4`, plus regression checks for canonical decimal IPv4 still passing.
- Live HTTP smoke: Transfer-Encoding rejected (400), 150-element batch rejected (-32600).

**Total assertions across all files**: 394 + 22 + 18 + 67 + 38 = **539** (was 519 at 1.9.3).

## Notes for follow-up

- **DNS resolution** for hostname classification (catches `127.0.0.1.nip.io` and similar rebinding) — needs a cyrius `getaddrinfo` stub that doesn't exist. Production callers should pair bote with a network-policy egress block to RFC 1918 / link-local.
- **OAuth 2.1 / PKCE-S256 / JWT verifier** — bearer middleware substrate (1.9.0) is already in place; flows + verifier are scope for a later post-2.0 release.
- **Claims propagation to handlers** — handler signature ABI change; want to land it in 2.0 with the right shape.
- **WS per-connection arena allocator** — long-lived WS connections accumulate per-frame bump-allocator waste; needs `fl_free` upstream.
